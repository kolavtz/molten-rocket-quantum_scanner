QuantumShield Project Instructions

Purpose
-------
Provide clear, enforceable implementation rules for building and evolving QuantumShield as a data-first, API-driven, security-led cybersecurity platform.

Non-Negotiable Principles
-------------------------
- Real data only. No seeded/demo/fabricated rows, metrics, domains, or scan outcomes in production artifacts.
- Database is the source of truth. UI, charts, exports, and reports must be backed by persisted rows.
- Security and correctness over speed. Use validation, sanitization, and parameterized SQLAlchemy queries.
- API-first architecture. Frontend and integrations must consume backend APIs, not hardcoded values.
- Explicit empty states. If data is missing, return zeros or actionable messages (never fake numbers).

Actual API Implementation Instructions
-------------------------------------
Use this section as the contract for all API work.

1) Route design and ownership
- Keep controllers/blueprints thin; business logic belongs in `/src/services`.
- Maintain RESTful resource naming and predictable route groups.
- Keep existing route behavior backward compatible unless explicitly approved.

2) Minimum required endpoints
- `GET /dashboard/api/certificates/telemetry`
- `GET /dashboard/api/certificates/inventory`
- `GET /dashboard/api/certificates/weak`
- `GET /dashboard/api/certificates/distribution/tls`
- `GET /dashboard/api/certificates/distribution/keys`
- `GET /dashboard/api/certificates/distribution/ca`
- `GET /`
- `POST /scan`
- `GET /results/<scan_id>`
- `GET /cbom/<scan_id>`
- `GET /api/scan`
- `GET /dashboard-metrics`

3) Request validation rules
- Validate and normalize all scan targets.
- Reject suspicious inputs (SQL/code injection patterns, malformed hosts, invalid CIDR).
- Limit CIDR scans to `/24` maximum.
- Default allowed ports: `443`, `8443`, `636`, `993`, `995`, `465`, `5061`, `80`, `22`, `3306`, `5432`, `8080` (configurable only via environment/config).
- Enforce endpoint timeout <= `10s` for probe operations.

4) Response and error contract
- Return JSON for API routes with stable keys and deterministic types.
- For collection endpoints, support pagination (`page`, `per_page`) where result size can grow.
- Include explicit counts/metadata for list responses when relevant.
- Use proper HTTP status codes:
  - `200` success
  - `201` created
  - `400` invalid input
  - `401/403` auth/role failures
  - `404` missing resource
  - `409` conflict
  - `422` semantic validation errors
  - `500` unexpected server errors
- Error payloads must be explicit and actionable (clear message, optional details, trace-safe).

5) Persistence and telemetry contract
- Every scan produces persisted records linked by a single `scan_id`.
- Network/probe failures must be logged and persisted as error state; they must not silently disappear.
- Certificate telemetry must persist and expose: `issuer`, `subject`, `serial`, `valid_from`, `valid_until`, `fingerprint_sha256`, `tls_version`, `key_length`, `cipher_suite`, `ca`, `is_deleted`, and soft-delete metadata.
- `CertificateTelemetryService` must compute totals, expirations, distributions, weak crypto metrics, and `cert_issues_count` from DB rows.
- `AssetService` must integrate `weak_cryptography` and `cert_issues_count` from persisted data only.

6) Authentication and RBAC
- Support admin/operator role checks in API and server-rendered routes.
- Never hardcode credentials, secrets, or role bypasses in source.
- `DEBUG_LOGIN_BYPASS` is allowed only when explicitly enabled in `.env` for development.

7) Export/report API rules
- CBOM export must be generated on demand from DB rows and conform to CycloneDX 1.6.
- Include Annexure-A fields when available in persisted data.
- Reporting endpoints and schedules must be backed by persisted `report_schedules` metadata.

8) API documentation and tests are mandatory
- Any new/changed endpoint must update corresponding docs under `/docs`.
- Add or update targeted tests for:
  - request validation
  - success behavior
  - empty-state behavior
  - error paths
  - role-based access behavior
- Do not weaken tests to force a pass; fix implementation defects.

Platform and Environment Rules
------------------------------
1. Store secrets and DB config in `.env`; commit only variable names in `.env.sample`.
   - `DATABASE_URL` or `DB_HOST`, `DB_PORT`, `DB_USER`, `DB_PASS`, `DB_NAME`
   - `SECRET_KEY`
   - `DEBUG_LOGIN_BYPASS=False` by default
2. Default stack is Python 3.10+, Flask, SQLAlchemy, Alembic.
3. Keep service classes small, composable, and testable.
4. Use parameterized queries and avoid N+1 query patterns.

UI and Template Rules
---------------------
- Top navigation only (no sidebar as primary nav).
- Dark glassmorphism theme and responsive layouts.
- Frontend must consume API-backed data; never hardcode dashboard values.
- Templates must render explicit empty states, for example:
  ```
  {% if cards.total_assets %}
    {{ cards.total_assets }}
  {% else %}
    <span>No scans yet — run your first scan</span>
  {% endif %}
  ```

Scanner and Crypto Requirements
-------------------------------
- TLS probes must capture and persist protocol, cipher, key length, signature algorithm, chain, SANs, validity, and fingerprint.
- Maintain static algorithm lookup tables for PQC labeling (Quantum Safe vs Quantum Vulnerable) based on NIST FIPS 203/204/205.
- Static lookup tables are allowed; fabricated scan outcomes are not.

Implementation Workflow Requirements
------------------------------------
1. Clarify scope and acceptance criteria.
2. Implement in small, testable increments.
3. Run validation after every significant step (especially scan execution and persistence changes).
4. Fix failing tests by correcting implementation behavior.
5. Update docs/instructions when behavior or contracts change.
