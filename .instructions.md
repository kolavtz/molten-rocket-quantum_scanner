QuantumShield Project Instructions

Purpose

This instructions file documents rules and conventions to be enforced across the QuantumShield project so generated code, prompts, and contributors follow the same expectations established in the conversation and SRS.

Scope

These rules apply to all source files, templates, generated code, automation scripts, and prompts used to extend the project unless an explicit exception is recorded and approved.

Key Rules (enforced)

1. No dummy or seeded production-style data
- Absolutely no hard-coded example rows, placeholder domains (e.g., example.com), or seeded demo metrics in code, templates, or fixtures that are shipped in the repository.
- Any UI metric must be computed at request time from MySQL queries. If the database contains no data for a metric, the UI must display "0" or a clear empty-state message such as "No data yet — run a scan".

2. Database: MySQL via .env
- MySQL is the primary persistent store. Use SQLAlchemy ORM models and migrations (Alembic recommended).
- Database connection configuration must come from environment variables (.env), not hard-coded values. Include a .env.sample file with variable names only (no secrets):
  - DATABASE_URL or DB_HOST/DB_PORT/DB_USER/DB_PASS/DB_NAME
  - SECRET_KEY
  - DEBUG_LOGIN_BYPASS (default False)
  - OTHER feature flags as needed

3. Top header navigation, no sidebar
- The app must use a top navigation header (Home, Asset Inventory, Asset Discovery, CBOM, Posture of PQC, Cyber Rating, Reporting). Do not implement a left/right sidebar menu as the primary navigation.

4. UI & styling
- Dark glassmorphism theme, Inter font, responsive design.
- Charts use Chart.js and must pull data from server JSON endpoints or server-rendered JSON variables derived from DB queries.
- Templates must handle empty-data states gracefully and never show placeholder metrics.

5. Project structure and naming
- Recommended structure:
  /app
    /routes    # blueprint/route handlers
    /models    # SQLAlchemy models
    /services  # scanner, risk engine, label engine, cbom generator
    /templates
    /static/css
    /static/js
  /migrations  # Alembic
  config.py
  .env.sample
  requirements.txt
- Use clear, RESTful route names (e.g., GET /, POST /scan, GET /results/<scan_id>, GET /cbom/<scan_id>, GET /api/scan).

6. Authentication & roles
- Implement a minimal role concept (admin/operator) at code level.
- Do NOT commit real user credentials. Implement DEBUG_LOGIN_BYPASS (False by default) to allow temporary session login in dev when explicitly enabled in .env.sample.

7. Scanner constraints and behaviour
- Scanner service input must validate and normalize targets; reject suspicious/injection strings.
- CIDR scan limit: /24 maximum.
- Allowed ports: 443, 8443, 636, 993, 995, 465, 5061, 80, 22, 3306, 5432, 8080 (configurable but default to this set).
- Per-endpoint timeout must be conservative (no longer than 10 seconds per endpoint by default) and network errors must be logged and handled without aborting whole scans.
- For TLS endpoints, the scanner extracts protocol, cipher, key length, signature algorithm, certificate chain, validity dates, SANs, fingerprint and persists in MySQL.

8. Cryptography & PQC classification
- Maintain internal reference tables/constants (not fake scan outputs) to map algorithms to "Quantum Safe" or "Quantum Vulnerable" per NIST FIPS 203/204/205.
- All algorithm classifications may be defined as static lookup tables in code but must not be used to fabricate scan results.

9. CBOM & export
- CycloneDX 1.6 JSON generation must be produced on-demand from DB rows (cbom_items, certificates, endpoints). CBOM may be exported as a file but not be the primary persistence layer.
- Annexure-A fields must be included per SRS when available.

10. Metrics and reports
- All dashboard and report metrics must be calculated from persisted scan data.
- Report scheduling entries should be persisted (report_schedules table); actual cron/email delivery may be mocked or stored for future workers.

11. Commit message trailer
- When committing changes, always append the Co-authored-by trailer:
  Co-authored-by: Copilot <223556219+Copilot@users.noreply.github.com>

Developer / Generator Guidance

- Default stack in prompts: Python 3.10+, Flask + SQLAlchemy + Alembic. Replace Flask with FastAPI only where explicitly desired.
- Prefer small, testable service classes in /app/services (TlsScannerService, RiskEngine, LabelEngine, CbomGenerator).
- Keep controller logic thin; perform heavy computations in services or model helpers.
- Use parameterized SQLAlchemy queries and avoid ORM anti-patterns (N+1 queries) when computing aggregates for dashboards.
- Provide JSON endpoints for Chart.js (e.g., /api/dashboard-metrics) that return precomputed counts and series from DB.

Template / UI requirements

- Each template must include conditional checks and explicit empty-state markup. Example Jinja2 pattern:
  {% if cards.total_assets %}
    {{ cards.total_assets }}
  {% else %}
    <span>No scans yet — run your first scan</span>
  {% endif %}

- Never render numeric placeholders; default to 0 or clear action prompts.

Example prompt snippets to reuse with Claude Code or other generators

- "Generate Flask route GET / that queries MySQL for total_assets, quantum_safe_count, vulnerable_count, avg_pqc_score and renders a Jinja2 template; if no scans exist return zeros and a 'No scans yet' call-to-action." 

- "Create a TlsScannerService.scan(targets, ports) that validates inputs, limits CIDR to /24, performs port discovery on default ports, runs TLS handshakes with timeout 10s, extracts certificate metadata and persists results transactionally under a single scan_id using SQLAlchemy." 

- "Create Alembic migration to add 'labels' table (id, scan_id, label, checksum, valid_from, valid_to, created_at) and an index on scan_id."

2. When adding new pages, update this instructions file with page-specific rules (e.g., required charts and endpoints) so the instruction set grows with the project.
