1. What is the Quantum Threat?

Today's internet security relies on public-key cryptography — algorithms like RSA, ECDSA, and Diffie-Hellman that protect everything from HTTPS connections to digital signatures. These algorithms are secure because factoring large numbers or solving discrete logarithm problems is computationally infeasible for classical computers.

Quantum computers change this equation entirely. Using Shor's algorithm, a sufficiently powerful quantum computer can break RSA-2048 and ECC-256 in polynomial time — rendering current TLS certificates, VPN tunnels, and code-signing infrastructure completely insecure.

⚠ HARVEST NOW, DECRYPT LATER (HNDL)

Adversaries are already recording encrypted traffic today with plans to decrypt it once quantum computers become available. Data with long confidentiality requirements (medical records, state secrets, financial data) is especially at risk. This makes migration urgent — not something to defer until quantum computers are "ready."

While fault-tolerant quantum computers are estimated to be 5–15 years away, the migration to quantum-safe cryptography must begin now because:

  • Cryptographic migrations typically take 5–10 years for large organizations
  • Hardware security modules (HSMs) and embedded systems have even longer upgrade cycles
  • Regulatory compliance deadlines (CNSA 2.0) begin as early as 2025

2. Why Post-Quantum Cryptography (PQC)?

Post-Quantum Cryptography refers to cryptographic algorithms that are believed to be secure against both classical and quantum attacks. Unlike quantum cryptography (which uses quantum mechanics directly), PQC algorithms run on regular classical hardware — they just rely on mathematical problems that quantum computers cannot solve efficiently.

🔷 LATTICE-BASED

ML-KEM (Kyber), ML-DSA (Dilithium). Most efficient. Basis of FIPS 203 & 204.

🔶 HASH-BASED

SLH-DSA (SPHINCS+). Conservative choice. Basis of FIPS 205. Minimal assumptions.

🔻 CODE-BASED

HQC (Round 4 candidate). Long research history. Larger key sizes.

3. NIST PQC Standardization Framework

The National Institute of Standards and Technology (NIST) has been running a multi-round competition since 2016 to select and standardize post-quantum algorithms. In August 2024, NIST published the first three final standards:

STANDARD ALGORITHM PURPOSE FAMILY
FIPS 203 ML-KEM (Kyber) Key Encapsulation Lattice
FIPS 204 ML-DSA (Dilithium) Digital Signatures Lattice
FIPS 205 SLH-DSA (SPHINCS+) Digital Signatures Hash-based
ℹ WHAT THIS MEANS FOR YOU

QuantumShield checks your TLS endpoints against these standards. If your server uses RSA or ECDSA exclusively, it will be flagged as quantum-vulnerable. The scanner detects hybrid key exchanges (e.g., X25519Kyber768) and PQC-aware certificate chains to determine your readiness level.

4. How QuantumShield Works

The scanner follows a 10-step pipeline for every target:

  1. Network Discovery — Port sweep to find TLS-capable services
  2. TLS Handshake — Full handshake to extract ciphers, protocol version, certificates
  3. PQC Detection — Checks for hybrid key exchanges, PQC KEMs, and PQC signatures
  4. CBOM Generation — Builds a Cryptographic Bill of Materials (CycloneDX format)
  5. Validation — Evaluates against NIST FIPS 203/204/205 compliance criteria
  6. Certificate Labeling — Issues quantum-safe / quantum-vulnerable labels
  7. Recommendation Engine — Generates prioritized migration recommendations
  8. Report Export — JSON + MySQL storage for historical tracking
  9. Audit Logging — Tamper-evident blockchain-style event chain
  10. Dashboard Visualization — Charts, badges, and risk assessment

5. Getting Started (Self-Hosted)

Prerequisites

  • Python 3.11+
  • MySQL 8.0+ (or MariaDB 10.6+)
  • mkcert (for trusted local HTTPS)

Installation

# 1. Clone the repository
git clone https://github.com/your-org/quantum-scanner.git
cd quantum-scanner

# 2. Create virtual environment
python -m venv .venv
.venv\Scripts\activate   # Windows
source .venv/bin/activate  # Linux/macOS

# 3. Install dependencies
pip install -r requirements.txt

# 4. Configure environment
cp .env.example .env
# Edit .env with your MySQL credentials, admin password, etc.

# 5. Initialize database
mysql -u root -p < schema.sql

# 6. Generate trusted HTTPS certs
mkcert -install
mkdir certs
mkcert -key-file certs/key.pem -cert-file certs/cert.pem localhost 127.0.0.1

# 7. Run the application
python web/app.py

Production Deployment

# Linux production with Gunicorn + Nginx
pip install gunicorn
gunicorn --bind 0.0.0.0:5000 \
  --workers 4 \
  --certfile certs/cert.pem \
  --keyfile certs/key.pem \
  web.app:app

# Windows production with Waitress
pip install waitress
waitress-serve --host 0.0.0.0 --port 5000 web.app:app

6. API Reference

All API endpoints require an API key. Get yours from the Profile page. Pass the key via X-API-Key header or ?api_key= query parameter.

GET /api/scan?target=example.com

Run a TLS scan on the specified target. Returns JSON report.

curl -H "X-API-Key: qss_YOUR_KEY" \
     "https://localhost:5000/api/scan?target=google.com"

POST /api/scan

Submit scan target via JSON body. Ideal for CI/CD pipelines.

curl -X POST "https://localhost:5000/api/scan" \
     -H "Content-Type: application/json" \
     -H "X-API-Key: qss_YOUR_KEY" \
     -d '{"target":"example.com"}'

GET /api/scans

List all recent scan results.

GET /api/badge/<target>

Public SVG badge. Embed in your README or CI dashboard. No API key required.

7. Cryptographic Bill of Materials (CBOM)

A CBOM is an inventory of all cryptographic assets used by a system — similar to how a Software Bill of Materials (SBOM) lists software components. QuantumShield generates CBOMs in the CycloneDX 1.6 format, including:

  • All TLS certificates in the chain (root, intermediate, leaf)
  • Key exchange algorithms (RSA, ECDHE, X25519, Kyber)
  • Signature algorithms (RSA-SHA256, ECDSA-P256, ML-DSA-65)
  • Cipher suites and protocol versions
  • Quantum-safe readiness status per component

Download CBOMs from any scan result page. Use them for compliance reporting, audit trails, and migration planning.

8. FAQ

Is my data stored?

Scan results are stored in MySQL for historical tracking. All audit logs are tamper-evident using a blockchain-style hash chain. No external telemetry is collected.

Can QuantumShield scan internal servers?

Yes. Deploy it on-premise and scan internal hostnames, IP addresses, or CIDR ranges. The scanner connects directly via TCP — no cloud dependency required.

When should I start migrating to PQC?

Now. NIST recommends immediate adoption of hybrid key exchanges. For sensitive data with >10-year confidentiality requirements, migration should be treated as urgent due to the HNDL attack vector.

Does the scanner modify my server configuration?

No. QuantumShield is read-only. It performs passive TLS handshakes and does not modify any server settings. The recommendations are advisory — you must implement changes yourself.

How do I make my server quantum-safe?

Follow the recommendations generated after each scan. Common steps include:

  • Enable hybrid key exchange (X25519Kyber768) in your TLS library
  • Use OpenSSL 3.2+ or BoringSSL with OQS provider for PQC support
  • Configure nginx/Apache to prefer PQC cipher suites
  • Obtain certificates from a CA that supports PQC signatures
